  - 관리자가 고객 요청부터 개발, 전달, 유지보수까지 통합 관리할 수 있는 시스템 구축

## 3. 핵심 가치 제안

### 유저 문제 요약
- 중소기업 대상의 합리적이고 맞춤형 자동화 솔루션 부족
- 기존 직원에게 자동화 도구 학습과 구축을 맡기기 어려우며, 신규 인력 고용도 부담
- 프리랜서 의존 시 품질 편차 및 사후 유지보수 공백 발생
- 표준화된 업무 매뉴얼 부재와 신속한 대응 어려움으로 지속적인 자동화 유지 불가

### 해결 방식 요약
- 고객이 요청한 업무 프로세스 정보를 노션 DB에 안전하게 암호화 저장
- 전문가들이 진단 및 맞춤형 자동화 솔루션 개발
- 개발 산출물은 전용 .daval 파일로 전달하며, 고객은 본인만의 암호키를 통해 사이트에서 안전하게 솔루션 직접 운영
- 관리자 페이지에서 모든 고객 요청사항 및 진행중인 프로젝트를 통합관리하며, 견적 제안부터 결과 검수, 고객 지원까지 책임

### 차별점 / 기존 대안 대비 강점
- CIO 출신 대표와 자동화 전문가가 직접 진단하여 최적 솔루션 제안
- 프리랜서 대비 100% 플랫폼 책임 유지보수를 통해 기술 의존성과 인력 이탈 위험 최소화
- 고유한 암호화 방식과 .daval 파일 시스템으로 고객 자산의 안전성과 일관성 확보
- 통합된 관리자 페이지를 통한 체계적인 프로젝트 관리 및 고객 지원

## 7. 비기능 요구사항

### 성능
- **응답 시간**: 
  - 페이지 로드 시간 3초 이내
  - API 응답 시간 1초 이내
  - .daval 파일 업로드/복호화 시간 5초 이내 (파일 크기 10MB 기준)
- **처리량**: 
  - 동시 접속 사용자 100명 지원
  - 초당 50개 API 요청 처리 가능

### 보안
- **인증 및 권한**:
  - JWT 기반 인증 시스템 구현
  - 역할 기반 접근 제어 (RBAC): 고객/관리자 권한 분리
  - 세션 타임아웃: 30분 무활동 시 자동 로그아웃
- **데이터 보호**:
  - 모든 민감 데이터 AES-256 암호화
  - HTTPS 통신 필수
  - 암호키는 환경변수로 관리 (절대 코드에 하드코딩 금지)
  - .daval 파일 고유 암호화 알고리즘 적용
  - 노션 API 통신 시 암호화 레이어 추가
- **개인정보 보호**:
  - 비밀번호 bcrypt 해싱 (최소 10 rounds)
  - 개인정보 처리 동의 필수

- **에러 처리**:
  - 모든 API에 try-catch 에러 핸들링 구현

  - React 컴포넌트 모듈화 및 재사용성 극대화
  - Node.js API 레이어 분리 (컨트롤러/서비스/레포지토리 패턴)
  - Vercel 서버리스 함수 활용

  - 노션 DB를 메인 데이터 저장소로 활용

- **관리자 대시보드 우선순위**:
  - 긴급도 높은 요청 우선 표시
  - 진행 중인 프로젝트 > 신규 요청 > 완료된 프로젝트 순

### 권한/역할 정책
- **고객(Customer)**:
  - 본인이 작성한 요청사항만 조회/수정 가능
  - 본인의 .daval 파일만 업로드/실행 가능
  - 본인의 암호키만 조회 가능
  - 다른 고객의 정보 접근 불가
- **관리자(Administrator)**:
  - 모든 고객 요청사항 조회/수정 권한
  - 모든 .daval 파일 생성/관리 권한
  - 모든 암호키 발급/관리 권한
  - 시스템 전체 모니터링 권한

### 기타 핵심 정책

**1. .daval 파일 정책**
- 파일 형식: 커스텀 .daval 확장자 (JSON 기반 암호화)
- 파일 크기 제한: 최대 50MB
- 파일 유효기간: 없음 (고객이 영구 보관 가능)
- 파일 버전 관리: 업데이트 시 새 버전 생성

**2. 암호키 관리 정책**
- 암호키 형식: 32자리 영숫자 조합
- 암호키 저장: 고객별 개별 암호화하여 DB 저장
- 암호키 분실 시: 본인 인증 후 재발급 가능 (이메일/SMS 인증)
- 암호키 변경 주기: 6개월마다 자동 갱신 알림 (선택적)

**3. 견적 및 결제 정책**
- 견적 유효기간: 제안 후 14일
- 견적 승인 방식: 고객 대시보드에서 원클릭 승인
- 결제 방식: 초기 버전은 오프라인 결제 (추후 온라인 통합)

**4. 유지보수 정책**
- 유지보수 대응 시간: 영업일 기준 24시간 이내
- 긴급 장애 대응: 2시간 이내
- 정기 점검: 월 1회

## 10. 리스크 & 오픈 이슈

### 주요 리스크

**기술적 리스크:**

1. **노션 DB 의존성**
   - 리스크: 노션 API 장애 시 서비스 전체 중단 가능
   - 완화 방안:

     - 노션 API 타임아웃 설정 및 재시도 로직
     - 대체 데이터 저장소 마이그레이션 계획 수립

2. **암호화 키 관리**
   - 리스크: 암호키 유출 시 고객 데이터 전체 노출
   - 완화 방안:
     - 암호키 별도 보안 스토리지 사용 (환경변수 암호화)
     - 암호키 접근 로그 철저히 관리
     - 정기적인 보안 감사

   - 리스크: 커스텀 포맷의 호환성 문제 및 버전 관리 어려움
   - 완화 방안:
     - 파일 버전 관리 시스템 구축
     - 하위 호환성 보장 정책 수립
     - 파일 검증 로직 강화

4. **Vercel 배포 제약사항**
   - 리스크: 서버리스 함수 실행 시간 제한 (10초), 메모리 제약
   - 완화 방안:
     - 장시간 작업은 백그라운드 작업으로 분리
     - Vercel Edge Functions 활용
     - 필요 시 외부 워커 서비스 연동

**비즈니스적 리스크:**

1. **전문가 의존성**
   - 리스크: CIO 대표 및 전문가 부재 시 진단 및 개발 지연
   - 완화 방안:
     - 진단 및 개발 프로세스 표준화 매뉴얼 작성
     - 백업 전문가 인력 확보
     - 단계별 체크리스트 시스템화

2. **초기 고객 확보**
   - 리스크: 신규 서비스로 초기 고객 유치 어려움
   - 완화 방안:
     - 파일럿 고객 프로그램 운영
     - 초기 할인 프로모션
     - 케이스 스터디 및 레퍼런스 확보

3. **고객 데이터 보안 사고**
   - 리스크: 데이터 유출 시 신뢰도 하락 및 법적 문제
   - 완화 방안:
     - 보안 인증 취득 (ISO 27001 등)
     - 보험 가입
     - 정기적인 보안 점검 및 침투 테스트

### 아직 결정되지 않은 사항

1. **결제 시스템 통합 시기**
   - 초기 버전은 오프라인 결제로 진행하지만, 온라인 결제 게이트웨이 통합 시기 미정
   - 토스페이먼츠, 이니시스 등 결제 서비스 선택 필요

2. **노션 DB 스키마 상세 설계**
   - 노션 데이터베이스의 정확한 필드 구조 및 관계 설정 필요
   - 암호화된 데이터를 노션에서 어떻게 저장할지 구체적 방안 논의 필요
   - **질문**: 노션 DB에 암호화된 데이터를 텍스트로 저장할지, 파일 첨부 형태로 저장할지 결정 필요

3. **.daval 파일 구조 및 암호화 알고리즘**
   - .daval 파일의 정확한 내부 구조 정의 필요
   - 사용할 암호화 알고리즘 (AES-256, RSA 등) 선택 필요
   - 파일 서명 및 무결성 검증 방식 결정 필요
   - **질문**: .daval 파일에 어떤 데이터가 포함되어야 하는지 구체적 명세 필요 (실행 가능한 코드? 설정 파일? 데이터?)

4. **자동화 솔루션 개발 범위**
   - 전문가가 개발하는 "자동화 솔루션"의 기술 스택 및 범위 미정
   - Zapier/Make.com 같은 도구 활용인지, 커스텀 스크립트 개발인지 명확화 필요
   - **질문**: 자동화 솔루션은 어떤 형태로 개발되나요? (예: Python 스크립트, Node.js 앱, 노코드 도구 워크플로우 등)

5. **고객-전문가 소통 방식**
   - 실시간 채팅, 이메일, 또는 플랫폼 내 메시징 시스템 중 선택 필요
   - **질문**: 고객과 전문가 간 소통은 어떤 채널로 진행하나요?

6. **모니터링 및 로깅 도구**
   - 시스템 모니터링 도구 선택 (Datadog, Sentry, LogRocket 등)
   - 에러 추적 및 성능 모니터링 도구 결정 필요

7. **백업 및 재해복구 계획**
   - 노션 DB 백업 주기 및 방법
   - 재해 발생 시 복구 시간 목표 (RTO) 및 복구 지점 목표 (RPO) 설정 필요

- **파일 구조**: 기능별로 폴더 구조화 (features-based structure)
